## 🎯 Северная звезда проекта

**Каждый проект сдан в срок с маржой 20–30%.**  
Все модули системы обслуживают эту цель: планировать, отслеживать, вовремя сигнализировать о рисках.

## Этап 3: Production Infrastructure & Testing

### Эпик 1: Миграция на PostgreSQL 15
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Перевод БД на PostgreSQL** | MUST HAVE | Alembic миграции. Все данные из SQLite перенесены. Row-Level Security по ролям (owner/accountant → full, manager → own projects, storekeeper → read). Пул соединений (asyncpg). |
| **Конфигурация окружений** | MUST HAVE | .env.staging / .env.production. Валидация переменных при старте (pydantic Settings). Никаких хардкод-секретов. ZAKUPPRO_API_KEY, DATABASE_URL, NEXTAUTH_SECRET — обязательны. |

### Эпик 2: Docker & Деплой
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Docker-композиция** | MUST HAVE | docker-compose.yml: app (Next.js) + api (FastAPI) + pg + nginx. Одна команда `docker compose up -d` — всё работает. Volumes для PG data и uploads. |
| **Деплой на Timeweb Cloud** | MUST HAVE | Dockerfile для каждого сервиса. nginx — reverse proxy + статика. Health check эндпоинты `/api/health`. Инструкция по деплою в README. |
| **CI/CD** | SHOULD HAVE | GitHub Actions: lint → test → build docker → push to registry → deploy webhook. Ручной approve для production. |

### Эпик 3: Тестирование
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Unit-тесты API** | MUST HAVE | Vitest + supertest. Покрытие всех edge cases из спецификации: amount ≤ 0 → 422, future date → 201 + warning, дедупликация 1C, revenue=0 → margin=0.0 (не NaN), tax_base < 0 → usn_tax = 0.0, checksum mismatch → ROLLBACK. Минимум 30 тестов. |
| **Integration-тесты** | SHOULD HAVE | Полный цикл: создать проект → добавить транзакции → проверить P&L → проверить бюджет → проверить cash flow. Тест с реальной PG (docker test container). |
| **E2E smoke** | SHOULD HAVE | Playwright: логин → дашборд → создать транзакцию → проверить в списке. 3-5 критических сценариев. |

## Этап 4: Операционная зрелость

### Эпик 4: Контроль маржинальности проектов (КЛЮЧЕВОЙ)
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Индикатор маржи на карточке проекта** | MUST HAVE | Цветовая индикация: 🟢 margin ≥ 30% — отлично, 🟡 20–29% — норма, 🠴 10–19% — внимание, 🔴 < 10% — критично. Отображается на дашборде и в списке проектов. |
| **Прогноз маржи по проекту** | MUST HAVE | Расчёт: (contractAmount − фактические расходы − плановые расходы) / contractAmount × 100%. Отображается рядом с фактической маржой. Позволяет видеть: «если потратим как запланировано — маржа будет X%». |
| **Алерт «Маржа падает»** | MUST HAVE | Уведомление при: (1) маржа проекта опустилась ниже 20%, (2) новая транзакция снизила маржу на > 5 п.п. за раз. Отправляется owner + manager проекта. |
| **Сравнение план/факт по проекту** | MUST HAVE | Таблица: категория | бюджет | факт | отклонение | % отклонения. Визуализация: bars бюджет vs факт. Drill-down по каждой категории. |
| **Дашборд «Проекты под угрозой»** | MUST HAVE | Отдельный блок на дашборде: проекты с маржой < 20% или с просроченным дедлайном. Сортировка по степени риска. |

### Эпик 5: Закрытие периода
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Закрытие месяца** | MUST HAVE | Owner/Accountant нажимает «Закрыть месяц» → все транзакции за этот месяц становятся read-only. Попытка редактирования → 403 + сообщение «Период закрыт». Запись в audit_log. |
| **Открытие периода (экстренное)** | SHOULD HAVE | Owner может reopen закрытый период с обязательным комментарием. Логируется. |

### Эпик 6: Полноценный 1C ClientBank
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Ручная загрузка** | MUST HAVE | Загрузка .txt файла через UI. Парсинг Win-1251, формат 1.05. Посимвольный разбор секций СекцияДокумент. Дедупликация по Номер+Дата. Автоклассификация. Флаг requires_classification. Отчёт об импорте: imported / duplicates / pending / errors. |
| **Автоматический импорт** | MUST HAVE | Watcher: мониторинг папки `/uploads/1c-incoming/` на появление новых файлов. При появлении → авто-импорт. Настройка пути к папке в .env. Лог каждого импорта в SyncLog. |
| **Автоклассификация** | MUST HAVE | Правила: (1) по контрагенту → known counterparty → категория из последней транзакции, (2) по regex в назначении платежа → r'проект\s*№\s*(ПМ\d+)', (3) по ключевым словам: «материал», «фурнитура», «аренд» и т.д. → соответствующая категория. Если ни одно правило не сработало → requires_classification = true. |
| **Ручная классификация** | MUST HAVE | Фильтр «Требуют классификации» в списке транзакций. Пакетное назначение проект+категория для выбранных транзакций. |

### Эпик 7: PDF-отчёты и автоматизация
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **P&L в PDF** | MUST HAVE | Шаблон: шапка ООО «ПРО Мебель», ИНН, период, таблица P&L, подпись. Генерация на backend (ReportLab/weasyprint). Скачивание из UI. |

### Эпик 8: UI/UX для продакшена
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Мобильная/планшетная версия** | MUST HAVE | Sidebar → bottom nav на мобильных. Крупные кнопки для кладовщика. Читаемые таблицы с горизонтальным скроллом. Минимум 3 breakpoint: mobile (<768), tablet (768-1024), desktop (>1024). |

## Этап 5: Hardening & Launch

### Эпик 9: Безопасность
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **HTTPS** | MUST HAVE | Nginx + Let's Encrypt (certbot). Автообновление сертификата. Редирект HTTP → HTTPS. |
| **Rate limiting** | MUST HAVE | Nginx: 100 req/min по IP, 1000 req/min по API-key. 429 при превышении. |
| **Защита от CSRF/XSS** | MUST HAVE | Next.js CSRF токены. Sanitization ввода. CSP заголовки. HttpOnly cookies для refresh токенов. |
| **Ротация API-ключей** | SHOULD HAVE | Генерация нового ключа без обрыва сервиса. Старый ключ живёт 24ч после замены. |

### Эпик 10: Отказоустойчивость
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **Бэкапы PostgreSQL** | MUST HAVE | Daily pg_dump → gzip → хранение 30 дней. Скрипт восстановления. Проверка бэкапа (тестовый restore раз в неделю). |
| **Health monitoring** | MUST HAVE | `/api/health` → { db: ok, zakuppro: ok, uptime, version }. Алерт при недоступности > 1 мин (email/webhook). |
| **Структурированные логи** | MUST HAVE | JSON-логи (pino/winston). Ротация по размеру (100MB). Уровни: error, warn, info, debug. В проде — info+. |
| **Graceful shutdown** | SHOULD HAVE | SIGTERM → завершить текущие запросы (10с таймаут) → закрыть DB pool → exit. |

### Эпик 11: Документация и запуск
| User Story | Приоритет | Критерии приемки |
|------------|-----------|-------------------|
| **API документация** | MUST HAVE | Swagger/OpenAPI автогенерация из FastAPI. Доступно по /docs. Все эндпоинты с примерами. |
| **Пользовательская инструкция** | MUST HAVE | Краткий гайд для каждой роли: Собственник (что смотреть), Бухгалтер (как закрывать период), Менеджер (как вводить транзакции), Кладовщик (как смотреть остатки). PDF или wiki. |
| **Runbook для администратора** | SHOULD HAVE | Деплой, бэкап/рестор, ротация ключей, мониторинг, экстренное открытие периода. |

## 📊 Приоритизация по бизнес-ценности

| Приоритет | Модуль | Почему |
|-----------|--------|--------|
| 🔴 P0 | Контроль маржи (Эпик 4) | Ядро ценности — видеть маржу в реальном времени |
| 🔴 P0 | 1C ClientBank реальный (Эпик 6) | Без импорта нет данных → нет маржи |
| 🔴 P0 | PostgreSQL + Docker (Эпики 1-2) | Без PG нет RLS, нет продакшена |
| 🟡 P1 | Закрытие периода (Эпик 5) | Бухгалтер не может работать без защиты данных |
| 🟡 P1 | PDF-отчёты (Эпик 7) | Руководство хочет видеть бумагу |
| 🟡 P1 | Тесты (Эпик 3) | Не сломать маржу при обновлениях |
| 🟢 P2 | Мобильная версия (Эпик 8) | Кладовщик вторичен, но нужен |
| 🟢 P2 | HTTPS + Rate limit (Эпик 9) | Для внутреннего использования терпит |
| 🟢 P2 | Бэкапы + мониторинг (Эпик 10) | После запуска, не до |
| 🟢 P2 | Документация (Эпик 11) | Параллельно с разработкой |

## 🏗️ Архитектурные решения

1. **FastAPI backend** — отдельный сервис от Next.js frontend. Next.js только рендерит UI, вся бизнес-логика и БД — в FastAPI. Это позволит независимо масштабировать и деплоить.
2. **Alembic** — миграции БД. Каждое изменение схемы — отдельная миграция с upgrade/downgrade.
3. **Redis** — кэширование P&L отчётов (инвалидация при новой транзакции). Опционально, добавляется при необходимости.
4. **Папка 1C incoming** — монтируется как Docker volume. FastAPI watcher следит за появлением файлов.
5. **Timeweb Cloud** — деплой через docker-compose. Один сервер (2 vCPU, 4GB RAM достаточно для 10 пользователей).
